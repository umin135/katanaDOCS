# 03. g1m 컨테이너와 청크 구조

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].

`g1m`은 하나의 3D 모델을 담는 **청크 컨테이너**다. 스켈레톤·역바인드행렬·지오메트리·물리(클로스/소프트) 등이 각각 청크로 들어간다.

---

## 1. 파일 구조 [실측]

```
g1m 파일
 ├─ 파일 헤더 (매직/버전/청크 수 등)
 └─ 청크 N개  (각각: 시그니처 + 버전 + 크기 + inner 데이터)
```

- 모든 정수는 **리틀엔디안**(단, 청크 버전 필드는 빅엔디안 4바이트로 읽히는 경우가 있어 파서가 별도 처리). [실측]
- 각 청크는 **시그니처(4바이트 ASCII)**, **버전**, **크기**, 그리고 **inner(실제 페이로드)**로 구성된다.
- **버전 예: `0x39323030`** = ASCII `"9200"` (DOA6 계열에서 관측). [실측] 버전은 일부 청크(특히 NUNO)의 내부 패딩 크기를 좌우한다.

---

## 2. 주요 청크 목록 [실측/유도]

| 시그니처 | 매직(u32 BE) | 역할 | 상세 문서 |
|---|---|---|---|
| **G1MF** | 0x47314D46 | 파일 정보 = **각종 카운트 테이블**(정점수·서브메시수·본맵수·NUNO 카운트 등). 다른 청크에 데이터를 추가하면 여기 카운트를 갱신해야 한다. | 본 문서 §3 |
| **G1MS** | 0x47314D53 | **스켈레톤**: 본 계층 + **jil**(로컬↔글로벌 ID 매핑). | [04](04_g1m_skeleton_and_matrices.md) |
| **G1MM** | 0x47314D4D | **역바인드행렬(inverse bind matrix)** 배열. 스키닝에 사용. G1MMIndex로 색인. | [04](04_g1m_skeleton_and_matrices.md) |
| **G1MG** | 0x47314D47 | **지오메트리**: 정점버퍼/레이아웃/인덱스/서브메시/조인트팔레트/메시그룹 등 여러 섹션의 묶음. | [05](05_g1m_geometry.md) |
| **NUNO** | 0x4E554E4F | **클로스(천) 시뮬**: NUNO1/NUNO3(제어점 격자), NUNO4(드라이버 테이블), NUNV(정점 클로스). | [06](06_g1m_cloth_system.md) |
| **SOFT** | (soft) | **소프트바디 물리**(예: 가슴 jiggle). 노드 격자 + 파라미터. | [08](08_g1m_textures_materials_physics.md) |
| (기타) | — | EXTR(외부 참조), COLL(콜리전) 등 [가설: 세부 미검증] | — |

주: 위 매직은 4바이트 ASCII를 빅엔디안 u32로 읽은 값이다. 파일 상에서는 `G1MG`처럼 ASCII로 보인다.

## 3. G1MF — 카운트 테이블 [실측]

G1MF는 g1m 전체의 여러 **개수(count)** 값을 담는 평평한 구조다. **어떤 청크에 엔트리/데이터를 추가하면, 대응하는 G1MF 카운트를 반드시 +갱신**해야 게임이 정상 로드한다.

### 3.1 오프셋 규약 [실측]
- G1MF의 필드 오프셋을 논할 때, **"구조체 오프셋(struct)"과 "inner 오프셋"이 0x0C 만큼 다르다.**
  - `inner_offset = struct_offset − 0x0C`
  - (구조체는 signature@0x00·version@0x04·size@0x08·unk@0x0C 를 포함하지만, inner는 그 12바이트 헤더를 뺀 지점부터 시작하기 때문.)
- 아래 표는 **inner 오프셋** 기준이다(코드에서 실제로 쓰는 값).

### 3.2 실측된 주요 G1MF 필드 (inner 오프셋)

| inner off | 필드 | 의미 | 추가 시 갱신 |
|---|---|---|---|
| 0x28 | num_vb | 정점버퍼 수 | VB 추가 시 +1 |
| 0x2C | num_layouts | 레이아웃 수 | +1 |
| 0x30 | num_layout_refs | 레이아웃 참조 수 | +1 |
| **0x34** | **num_bone_maps** | **조인트 팔레트(본맵) 수 = 0x10006 섹션의 npal** | 팔레트 추가 시 +추가수 |
| **0x38** | **num_individual_bone_maps** | **전 팔레트의 총 엔트리 수** | +추가 엔트리 총합 |
| 0x3C | (num_layout_refs 계열) | | |
| 0x40 | num_submeshes | 서브메시 수 | +1 |
| 0x44 | num_submeshes2 | 서브메시 수(2) | +1 |
| 0x4C | num_meshes | 메시그룹 엔트리 수 | +1 |
| 0x50 | num_submeshes_in_meshes | | +1 |
| **0x68** | **num_nuno1s** | NUNO1 엔트리 수 | NUNO1 추가 시 +1 |
| 0x6C | num_nuno1s_unk4 | (엔트리당) | +1 |
| 0x70 | num_nuno1s_control_points | NUNO1 제어점 총합 | += cpc |
| 0x74 | num_nuno1s_unk1 | NUNO1 unkcnt(앵커) 총합 | += unkcnt |
| 0x78 | num_nuno1s_unk2_and_unk3 | NUNO1 skip1+skip2 총합 | += (skip1+skip2) |
| **0xE4** | **num_nuno3s** | NUNO3 엔트리 수 | NUNO3 추가 시 +1 |
| 0xE8 | num_nuno3s_unk4 | (엔트리당) | +1 |
| 0xEC | num_nuno3s_control_points | NUNO3 제어점 총합 | += cpc |
| 0xF0 | num_nuno3s_unk1 | NUNO3 unkSection 총합 | += unkcnt |

> **주의 [실측]:** "값이 npal과 같은 필드를 검색해서 +1" 같은 방식은 위험하다. 예컨대 num_bone_maps(0x34) 값이 num_vb·num_submeshes 등 여러 필드와 **우연히 같을 수 있어**, 검색 방식은 엉뚱한 필드를 건드린다. **반드시 정확한 오프셋을 직접 써라.**

## 4. 청크 조작 시 안전 규칙 [유도]

- **엔트리는 섹션/청크의 "끝"에 삽입**하라. 중간 삽입은 인덱스 기반 참조(ext_id, 팔레트 인덱스 등)를 깨뜨린다. [실측]
  - 인덱스 기반 참조들은 "n번째 엔트리"로 가리키므로, 끝에 추가하면 기존 인덱스가 보존된다.
- 청크 내부에 데이터를 삽입하면 **그 청크의 크기 필드와 엔트리 카운트**를 갱신하고, **G1MF의 대응 카운트**도 갱신한다.
- 원본은 **인플레이스 수정 금지**, 별도 출력 + 백업(`.kmbak`), 항상 **깨끗한 원본에서 멱등 재계산**.
