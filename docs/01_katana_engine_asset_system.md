# 01. KatanaEngine 에셋 시스템 (RDB / RDX / FDATA)

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].

KatanaEngine(코에이 테크모)은 게임 에셋을 **인덱스 계층**과 **데이터 계층** 두 층으로 관리한다. 이 문서는 그 두 층의 바이너리 포맷과, 에셋 하나가 로드되는 흐름을 다룬다.

---

## 1. 두 계층 개요 [실측]

```
[인덱스 계층]
  root.rdb   : file_ktid → (fdata_id, offset, size) 등 위치 메타
  root.rdx   : fdata_id  → 파일명 해시 (파일명 해석용)

[데이터 계층]
  0x????????.fdata : IDRK 블록들의 컨테이너 (zlibext 압축된 에셋 페이로드)
```

- **인덱스(RDB/RDX)** 는 "어떤 에셋이 어느 .fdata의 어디에 있는가"를 안다.
- **데이터(FDATA)** 는 실제 에셋 바이트(모델·텍스처·재질 등)를 압축 블록으로 담는다.

여러 .fdata 파일이 존재하며, 각각 16비트 ID(`fdata_id`, 0~65535)로 식별된다. [실측]

---

## 2. 에셋 로드 흐름 [유도]

게임이 에셋 하나를 로드하는 논리적 흐름:

```
LoadAsset(file_ktid):
  1) RDB에서 file_ktid로 엔트리 검색 → (fdata_id, offset, size) 획득
  2) RDX로 fdata_id → 파일명(해시) 해석 → 0x????????.fdata 파일 결정
  3) 그 .fdata의 offset 위치에서 IDRK 블록 읽기
  4) IDRK 페이로드를 zlibext 해제 → 원본 에셋 바이트
```

이 흐름 때문에, **원본 .fdata를 건드리지 않고도** RDB 엔트리의 위치 메타만 다른 .fdata로 돌리면(리다이렉트) 에셋을 교체할 수 있다(→ §6 오버라이드 전략).

---

## 3. RDB / RDX 포맷 [실측]

### 3.1 공통
- 모든 정수는 **리틀엔디안**.
- RDB 엔트리는 **4바이트 정렬**된다(stride가 4의 배수).

### 3.2 RDB (root.rdb)
- 헤더에 **`file_count`(엔트리 수)** 필드가 있다. 엔트리를 추가하면 이 값을 갱신해야 한다. [실측]
- 각 엔트리는 하나의 에셋에 대한 위치 메타를 담는다. 관측된 위치 인코딩은 두 종류가 있다:
  - **Location32** — 32비트 위치 필드 계열
  - **Location40** — 40비트 위치 필드 계열
  (게임/버전에 따라 어느 인코딩인지 판별해서 파싱한다.)
- 엔트리에는 대략 다음이 담긴다: **file_ktid**(에셋 고유 해시), **type_ktid**(타입 클래스 해시), **fdata_id**(u16, 어느 .fdata인지), **offset/size**(그 .fdata 내 위치·크기), 그 외 플래그.
- **바이너리 안전 규칙 [유도]:** 엔트리를 **수정**할 때 `entry_size`(엔트리 크기 필드)를 바꾸지 마라. 내부 필드만 덮어써라. 엔트리를 **추가**할 때만 헤더 `file_count`를 갱신한다.

### 3.3 RDX (root.rdx)
- **RDX에는 별도 헤더가 없다.** [실측] (RDB와 달리 바로 매핑 데이터가 온다.)
- `fdata_id` → 파일명 해시 매핑을 담는다. 이 해시로 실제 `0x????????.fdata` 파일명이 결정된다.

---

## 4. FDATA / IDRK 포맷 [실측]

`.fdata` 파일은 **IDRK 블록**들의 연속이다.

### 4.1 IDRK 블록 구조
- 각 블록은 매직 `IDRK`로 시작한다.
- 블록 헤더에는 크기 필드들과 `param` 필드가 있다.
- **★가장 중요한 실측 사실:** **페이로드(압축 데이터)의 위치는 "블록 끝에서 compressed_size 만큼 앞"으로 계산해야 한다.**
  - `param` 필드로 페이로드 오프셋을 계산하면 **틀린다.** (param은 페이로드 시작을 직접 가리키지 않는다.) [실측]
  - 즉 `payload_start = block_end - compressed_size`.

### 4.2 zlibext 압축
- IDRK 페이로드는 **완전한 zlib 스트림**(zlibext)이다. 표준 zlib inflate로 해제된다. [실측]

### 4.3 무결성 검증 없음
- **엔진은 IDRK 블록의 청크 해시/CRC 무결성을 검증하지 않는다.** [실측, DOA6 LR]
  - 따라서 에셋 바이트를 바꿔도 체크섬을 재계산할 필요가 없다.
  - 이 사실이 모드 제작을 크게 단순화한다.

---

## 5. KTID (에셋 해시) [실측/미해결]

- **FileKtid** — 에셋 고유 32비트 해시. 원본 **파일명 기반**으로 만들어진다.
- **TypeKtid** — 타입 클래스 해시. TypeKtid로 확장자/에셋 종류를 판별한다(TypeKtid→확장자 매핑 테이블 존재).

### 5.1 네임해시 문제 [미해결]
- **파일명 → FileKtid 해시 알고리즘은 완전히 규명되지 못했다.** 관측 결과 단순 선형/CRC류가 아니라 비선형으로 보인다. [실측]
- 우회책: 이미 알려진 **파일명↔FileKtid 대응표(CSV 등)**를 확보해 두고, 새 해시를 크래킹하지 않고 표로 조회한다. [유도]
- 새 에셋을 "새 파일명으로" 추가하려면 이 해시가 필요하지만, 기존 에셋을 리다이렉트/교체하는 데는 필요 없다(기존 FileKtid를 재사용).

---

## 6. 오버라이드(교체) 전략 [실측/가설]

에셋을 교체하는 4가지 접근이 검토되었다. 핵심은 **원본 파괴를 피하는 것**이다.

| 전략 | 방식 | 상태 |
|---|---|---|
| **A** | 원본 .fdata 인플레이스 수정 | 파괴적 → 지양 |
| **B** | **RDB 엔트리 리다이렉트** — 원본은 그대로 두고, RDB의 위치 메타를 커스텀 .fdata로 돌림 | **[실측] DOA6 LR에서 검증 완료** |
| **C** | last-wins 중복 추가 — 같은 KTID 엔트리를 뒤에 또 추가하면 나중 것이 이김 | [가설] 미검증(B가 확정돼 검증 불요) |
| **D** | 기타 | 미탐색 |

### 6.1 전략 B (권장) 절차 [실측]
1. 커스텀 에셋 바이트를 zlibext로 압축해 **IDRK 블록**으로 만든다.
2. 그 블록을 **새 .fdata**(또는 mods.fdata)에 담는다.
3. 새 .fdata에 **미사용 fdata_id(u16)**를 할당하고 RDX/시스템에 등록한다.
4. 교체할 에셋의 **RDB 엔트리를 수정**해 (fdata_id, offset, size)를 새 블록으로 돌린다. (`entry_size`는 건드리지 않는다.)
5. 원본 .fdata·RDB는 백업(예: `.kmbak`) 후 항상 **깨끗한 원본에서 재계산(멱등)**으로 패치한다.

- **검증 결과 [실측]:** DOA6 LR에서 g1t 텍스처를 커스텀 mods.fdata로 리다이렉트 → 게임 내 반영 확인. 마젠타 테스트 텍스처로 실측.

---

## 7. 안전 규칙 요약 [유도]

- 정수는 전부 **리틀엔디안**, RDB 엔트리는 **4바이트 정렬**.
- 원본(rdb/rdx/fdata)은 **인플레이스 수정 금지**. 패치본은 별도 출력, 원본은 백업.
- 모드 적용은 항상 **깨끗한 원본에서 재계산(멱등)**.
- RDB 엔트리 **수정 시 entry_size 불변**, **추가 시 file_count 갱신**.
- `fdata_id`는 u16 → 새 .fdata 등록 시 **미사용 ID** 할당.
- IDRK 무결성 체크가 없으므로 체크섬 재계산 불필요.

---

## 8. 이 계층과 g1m의 관계

RDB/RDX/FDATA는 **에셋을 담고 찾는 그릇**이다. 그 안에 담기는 대표 에셋이 **g1m(3D 모델)**, **g1t(텍스처)**, **kts(재질)** 등이다. 한 캐릭터 복장이 이 에셋들을 어떻게 서로 참조하는지는 [02_asset_reference_chain.md](02_asset_reference_chain.md)에서, g1m 자체의 내부 구조는 [03_g1m_container_and_chunks.md](03_g1m_container_and_chunks.md)부터 다룬다.
