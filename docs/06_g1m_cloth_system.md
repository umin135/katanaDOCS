# 06. g1m 클로스(NUNO) 시스템 — 심층

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].
> **이 문서 세트의 핵심.** 가장 많은 실측·시행착오가 여기에 있다.

**NUNO** 청크는 g1m의 **천(cloth) 시뮬레이션**을 담는다. 이 문서는 그 바이너리 구조와, "왜 클로스가 그렇게 동작하는가"의 forward model, 그리고 우리를 오래 헤매게 한 **스파이크**·**엉뚱한 위치**·**흔들림 없음**의 원인을 전부 정리한다.

---

## 1. NUNO 청크 개요 [실측]

NUNO 청크는 섹션들의 묶음이다:

```
NUNO inner:
  section_count (u32)
  반복: (magic u32, chunk_size u32, entry_count u32) + 엔트리들
```

| magic | 이름 | 역할 |
|---|---|---|
| 0x00030001 | **NUNO1** | 제어점 격자(클로스). **디스플레이 없음(물리 격자만).** |
| 0x00030003 | **NUNO3** | 제어점 격자(클로스). **디스플레이 메시가 연결됨(ext=20000+idx).** |
| 0x00030004 | **NUNO4** | 드라이버 본 테이블(로컬↔글로벌 매핑, 클로스 드라이버 체인). |
| 0x00010001 | **NUNV** | 정점 기반 클로스(다른 종류). |
| 0x00030005 | NUNO5 | 신형(Ryza/Yumia 계열). DOA6엔 없음. |

## 2. ★모든 클로스 = NUNO1 + NUNO3 쌍 [실측]

**실제 게임의 모든 클로스는 같은 격자를 NUNO1과 NUNO3 두 섹션에 중복 저장한다.**

- 실측: 한 도너 클로스의 NUNO1[0]과 NUNO3[0]은 **제어점 98/98 완전 동일**, **influences(P1~P4 위상 + P5/P6 rest-length) 98/98 완전 동일**. 같은 parentID.
- KOK 손목 클로스: NUNO1[75,76] + NUNO3[75,76] (좌우 손목 각각 쌍).
- RYU 스커트: NUNO1 2개 + NUNO3 2개.

즉 **CP·influences는 두 섹션이 똑같고, 다른 것은 "skip 구조"뿐**이며, 그 skip 구조는 격자의 **rows/cols에서 파생**된다(§5). 그래서 저작 시에는 **격자 하나(rows/cols/CP)만 있으면 NUNO1·NUNO3를 둘 다 생성**할 수 있다.

- **디스플레이 메시는 NUNO3에만 연결**(ext=20000+idx)된다. NUNO1은 디스플레이 없이 물리 격자로만 존재한다. [실측]
- **NUNO1의 정확한 게임 내 역할(왜 중복 저장하는가)은 완전히 규명되지 않았다** — 물리 시뮬 격자/레거시 이중솔버로 추정. [가설] 다만 **실제 클로스는 항상 쌍을 갖추므로, 저작 시에도 쌍을 만드는 것이 안전**하다. (NUNO3 단독으로도 시뮬 자체는 도는 것을 실측했으나, 원본 구조 일치를 위해 쌍을 권장.)

## 3. NUNO3 엔트리 구조 [실측]

버전 `"9200"`(0x39323030) 기준.

```
NUNO3 엔트리:
  [헤더 32바이트]
    +0x00  parentID  (u32)   # ★글로벌 본 ID (§8 forward model)
    +0x04  cpc       (u32)   # 제어점 수 (rows×cols)
    +0x08  unkcnt    (u32)   # 앵커(unkSection) 수 = 2×cols (상단 2행)
    +0x0C  skip1     (u32)
    +0x10  [04]/refBone (u32) # ★참조/기준 본 (아래 §7 U_10)
    +0x14  skip2     (u32)
    +0x18  skip3     (u32)   # tail 링크(테더) 수 = (rows-2)×cols
    +0x1C  skip4     (u32)
  [head 파라미터]  +0x20 ~ +0xD7  (버전에 따라 0xA8+0x10 패딩; §7)
  [제어점]  cpc × 16바이트 (RichVec4: x,y,z,w)
  [influences]  cpc × 24바이트 (NunInfluence)
  [unkSection(앵커)]  unkcnt × 48바이트
  [skips]  (4×skip1 + 8×skip2 + 12×skip3 + 8×skip4) 바이트
```

- **CP 시작 오프셋 = 0xD8** (헤더 32 + head 0xA8 + 0x10). [실측]

## 4. 제어점·influences·앵커 [실측]

### 4.1 제어점 (Control Point, RichVec4)
- `(x, y, z, w)` 4 float, 16바이트.
- **위치는 드라이버 본의 로컬 프레임**에 저장된다(§8).
- `w`는 플래그/역질량 성격(관측: 리벳/시뮬 구분과 무관하게 1.0인 경우가 많음).

### 4.2 influences (NunInfluence, 24바이트)
```
struct NunInfluence {
  int   P1;   # 왼쪽 이웃 CP 인덱스 (-1=없음)
  int   P2;   # 오른쪽 이웃 CP 인덱스
  int   P3;   # 위(부모) 이웃 CP 인덱스 (-1=최상단 행=루트)
  int   P4;   # 아래(자식) 이웃 CP 인덱스
  float P5;   # rest-length (P2 이웃까지 거리)
  float P6;   # rest-length (P4 이웃까지 거리)
}
```
- **P1~P4 = 격자 위상(4방향 이웃)**. 이것이 클로스의 구속 그래프를 정의한다.
- **P3 = -1인 CP = 최상단 행(루트)** — 앵커에 고정되는 행. [실측]
- **P5/P6 = rest-length**(스프링 자연길이). 격자를 스케일/변형하면 재계산해야 한다. [실측]

### 4.3 앵커 (unkSection, 48바이트) [실측]
상단 2행(= 2×cols 개)의 제어점을 본에 고정한다.
```
앵커 48바이트:
  +0x00  방향벡터/데이터 (float ×8 = 32바이트)  # 노멀 등, 드라이버 본 프레임서 회전됨
  +0x20  count (u32)                            # 이 앵커의 본 가중 수
  +0x24  packedBoneIdx (u32)                    # ★3 × (팔레트 슬롯) — 리벳 i1과 동일 인코딩
```
- **앵커의 `@0x24`(packedBoneIdx)는 3×팔레트슬롯**이다. raw 로컬본을 쓰면 게임이 //3 오해석 → 상단↔본체 프레임 불일치로 찢어짐. [실측]
- 앵커의 방향벡터(@0x00)는 **드라이버 본 프레임에 맞춰 회전**되어 저장된다(축정렬 본이면 (0,0,-1)(0,1,0) 같은 깔끔한 값, 회전된 본이면 그에 맞게 기울어진 값). [실측]

## 5. NUNO1 엔트리 구조 & skip 파생 [실측]

NUNO1은 CP·influences·앵커가 NUNO3와 동일하되, 헤더/패딩/skip이 다르다.

```
NUNO1 엔트리 (ver 9200):
  [헤더 24바이트]
    +0x00 parentID, +0x04 cpc, +0x08 unkcnt,
    +0x0C skip1, +0x10 skip2, +0x14 skip3
  [pad 0x5C]   # 0x3C + 0x10(ver>0x30303233) + 0x10(ver>=0x30303235)
               # pad idx2(entry@0x20) = 참조 본 (NUNO3의 [04]와 동일하게)
  [제어점] cpc×16   [influences] cpc×24   [unkSection] 48×unkcnt
  [skips]  4×(skip1+skip2+skip3)
```

### 5.1 skip 값 = rows/cols에서 파생 [실측]
격자가 `rows × cols`(cpc = rows×cols, cols = unkcnt//2)일 때:

| skip | 개수 | 내용(인덱스 리스트) |
|---|---|---|
| skip1 | cols | 첫 행: `[0, 1, …, cols-1]` |
| skip2 | rows-2 | 2행+ 의 col0: `[r*cols for r in 2..rows-1]` |
| skip3 | 1 | `[2]` |

- 실측 예: cpc=98, unkcnt=14 → cols=7, rows=14. skip1=7([0..6]), skip2=12([14,21,…,91]), skip3=1([2]).
- **즉 격자 하나(rows/cols)면 NUNO1 skip을 전부 파생**할 수 있다 — 별도 데이터 불필요.

## 6. ★Forward Model — 시뮬 위치가 어떻게 정해지는가 [실측]

우리를 가장 오래 헤매게 한 부분. **CP의 월드 위치는 이렇게 정해진다:**

```
CP_world = world[ globalToFinal[parentID] ] · CP_local
```

- **저작(인코딩) 시**: CP를 `CP_local = inv(world[드라이버로컬본]) · CP_model` 로 저장한다. 즉 CP는 **드라이버 로컬 본의 프레임**에 담긴다.
- **게임(디코딩) 시**: `world[globalToFinal[parentID]] · CP_local` 로 월드에 배치한다.
- **두 프레임이 일치하려면**: `globalToFinal[parentID] = 드라이버로컬본`.
- 자기완결 g1m에서 `globalToFinal[l2g[X]] = X`([04] §3)이므로, **parentID = l2g[드라이버로컬본] = 글로벌 ID**여야 한다.

### 6.1 실측 검증 (KOK 손목 클로스) [실측]
- NUNO3 parentID = 75. g2l[75] = 로컬 bone68(RF_L_ForeArm_01).
- `world[bone68] · CP_local` 의 월드 중심 = **(57.8, 6.4, -2.6)**.
- 그 클로스의 **디스플레이 메시 중심 = (57.9, 6.9, -3.0)**. **거리 0.6** → 사실상 일치.
- 즉 forward model이 정확히 성립. 우리 파싱(g2l/l2g)도 정확함이 이로써 확인됨.

### 6.2 parentID를 "로컬"로 쓰면 생기는 버그 [실측]
- parentID = 로컬본(예 68)로 쓰면 → `globalToFinal[68]` = 글로벌 68인 **엉뚱한 본**(예: 허벅지) → CP 프레임 불일치 → **시뮬 정점이 엉뚱한 곳에서 덜덜 떨림**.
- (이 증상을 오래 "한계"로 오해했으나, 원인은 parentID 로컬/글로벌 혼동이었다.)

## 7. head 파라미터와 [04](U_10) [실측/미해결]

### 7.1 head 파라미터 (NUNO3 +0x20~0xD7)
- 강성(stiffness)·댐핑·기타 시뮬 파라미터가 float/flag로 들어있다.
- 관측: 같은 캐릭터의 두 클로스(좌/우)를 비교하면 **대부분의 head 필드는 동일**하고, **소수 필드만 클로스별로 다르다**(예: 강성 값 2개, 그리고 [04]).
- 저작 시에는 **도너(작동하는 원본) head를 통째로 복사**해 쓰는 것이 안전하다(강성 등 시뮬 파라미터 재현).

### 7.2 [04] = U_10 = 참조/기준 본 (NUNO3 +0x10) [실측/미해결]
- 이 필드는 **낮은 SK 본 인덱스**를 담는다. 관측 예:
  - 척추 클로스(SK_Spine02 구동) → [04]=SK_Spine01
  - 손목 클로스(RF_L_ForeArm 구동) → [04]=SK_L_Shoulder (팔의 SK 뿌리)
  - 스커트(SK_Spine02 구동) → [04]=SK_L_Leg (다리)
- **[04]는 반드시 "낮은 SK 본"이어야 한다.** 높은 RF 본(트위스트)을 넣으면 **게임이 크래시**한다. [실측]
- **[04]의 정확한 규칙과 시뮬에 미치는 영향은 미해결.** [미해결] 도너 값을 그대로 복사하면 위치가 어긋날 수 있다는 정황이 있었으나(척추 구동 vs 다리 위치), forward model(parentID=글로벌)을 바로잡은 뒤 재검증은 하지 못했다. 저작 후 인게임 테스트로 튜닝해야 한다.

## 8. ★클로스 팔레트(physicsIndex)와 스파이크 [실측]

**클로스가 시뮬되려면, 디스플레이 메시가 바인딩한 조인트 팔레트의 `physicsIndex`가 드라이버 본이어야 한다.**

### 8.1 정점 시맨틱: 리벳 vs 시뮬 [실측]
클로스 디스플레이 메시의 정점은 두 종류이고, **`cpW2`(정점버퍼 offset 32의 4 float) 플래그로 구별**된다:

| 정점 종류 | cpW2 | i1(블렌드 인덱스)의 의미 |
|---|---|---|
| **리벳(고정)** | = 0 | **팔레트 슬롯** (3×slot). 본에 직접 스킨. |
| **시뮬(자유)** | ≠ 0 | **제어점(CP)/드라이버 조인트 인덱스**. NUNO 물리에 구동. |

- 즉 시뮬 정점의 i1은 "CP 인덱스"이고, 게임은 이걸 **NUNO 드라이버 조인트**로 해석한다(팔레트 슬롯이 아님).

### 8.2 스파이크의 원인 [실측]
- 게임이 "이 메시는 물리(클로스) 메시"라고 인식해야, 시뮬 정점(cpW2≠0)의 i1을 CP 인덱스로 올바르게 해석한다.
- **이 인식은 디스플레이 메시가 바인딩한 팔레트의 `physicsIndex`로 이뤄진다.** `physicsIndex=드라이버본`이면 물리 메시로 인식.
- **`physicsIndex=0`(일반 팔레트)에 바인딩하면**, 게임이 일반 스키닝으로 오인 → 시뮬 정점의 i1(=CP 인덱스, 예 7~11)을 **팔레트 본 슬롯으로 오독** → 엉뚱한 본에 스키닝 → **정점 폭발(spike).** [실측, XML 도구로 확정]
- (초기 저작 도구가 기존 26본 `physIdx=0` 팔레트를 재사용한 것이 스파이크의 근본원인이었다.)

### 8.3 디스플레이 팔레트 vs 물리 팔레트 (2개 필요) [실측]
비-항등 본(로컬≠글로벌)에서는 **팔레트 2개**가 필요하다(실제 KOK 구조):

| 팔레트 | physicsIndex | jointIndex | 용도 |
|---|---|---|---|
| **디스플레이** | 로컬 드라이버본 | 로컬 본들 | **서브메시가 바인딩**. 리벳/앵커 슬롯이 여기 참조. |
| **물리** | **글로벌 드라이버본 (= parentID)** | 글로벌 본들 | 게임 물리 링크. 서브메시 바인딩 안 함. |

- **G1MMIndex는 두 팔레트 모두 드라이버 본의 것**(스키닝 행렬은 G1MMIndex로 결정되므로 로컬/글로벌 상관없이 같은 본 행렬).
- 항등 본(로컬=글로벌, 예 bone10)은 하나로 충분(physIdx=10이 로컬이자 글로벌).

## 9. ★jiggle 근본 제약 — 구동 본이 움직여야 한다 [실측]

- **클로스 시뮬의 눈에 보이는 흔들림은 구동 본(parentID)의 실제 움직임에서 나온다.**
- 실측:
  - bone10=SK_Spine02(호흡으로 항상 미세하게 움직임) 구동 → idle에도 jiggle.
  - RF_ForeArm(팔이 흔들림) 구동 → jiggle.
  - **RF_UpLeg(가만히 선 다리, 고정) 구동 → 다리를 움직여도 rigid**(흔들림 없음).
- 즉 **고정된 본에 직접 부착한 클로스는 흔들리지 않는다.** [실측]
- 실제 게임의 다리/스커트 클로스는 **허리(움직이는 SK_Spine02) 구동 + 다리 위로 드리움**(RYU 방식)으로 만든다. 다리 자체를 구동으로 쓰지 않는다.
- **함의:** 흔들리는 클로스를 원하면 **idle에도 움직이는 사지(손목/팔뚝) 또는 허리**를 드라이버로 삼아야 한다.

## 10. NUNO4 — 드라이버 본 테이블 [실측/가설]

- magic 0x00030004. 클로스 드라이버 본의 **로컬↔글로벌 매핑 테이블**(연속 클로스 드라이버 체인).
- 구조(관측): head[1]=본 수 n, tail에 (local, global) 쌍 n개.
- 앵커의 로컬 본 인덱스를 전역 스켈레톤 본으로 변환하는 용도로 관측됨. [가설: 세부 역할 미확정]
- 우리 저작 클로스(NUNO3 단독/쌍)는 새 NUNO4 없이도 동작을 관측했으므로, NUNO4는 도너 클로스 전용이거나 특정 기능용으로 추정. [가설]
