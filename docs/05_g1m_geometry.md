# 05. g1m 지오메트리 (G1MG · 정점버퍼 · 조인트 팔레트 · 메시그룹)

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].

**G1MG**는 실제 렌더 지오메트리를 담는 청크로, 여러 **섹션**의 묶음이다. 클로스 저작에 특히 중요한 것은 **서브메시**·**조인트 팔레트**·**메시그룹**이다.

---

## 1. G1MG 섹션 목록 [실측]

G1MG inner는 헤더 뒤에 섹션들이 온다. 각 섹션은 **섹션 ID**로 구분된다.

| 섹션 ID | 내용 |
|---|---|
| **0x10004** | **VertexBuffer** — 정점 raw 데이터(여러 VB) |
| **0x10005** | **Layout** — 정점 시맨틱 레이아웃(어떤 필드가 어느 오프셋/타입) |
| **0x10006** | **JointPalettes (본맵)** — 서브메시가 쓰는 (G1MMIndex, physicsIndex, jointIndex) 팔레트들 |
| **0x10007** | **IndexBuffer** — 삼각형 인덱스(여러 IB) |
| **0x10008** | **Submesh** — 서브메시 엔트리(스트라이드 0x38) |
| **0x10009** | **MeshGroups** — 서브메시 그룹화 + **ext_id(클로스/물리 연결)** |

---

## 2. Submesh (0x10008) [실측]

서브메시 엔트리는 **스트라이드 0x38바이트**, 섹션은 앞 4바이트(개수) 뒤에 엔트리들이 온다(`base=4`).

### 2.1 실측된 엔트리 필드 (엔트리 내 오프셋)
| off | 필드 | 의미 |
|---|---|---|
| 0x04 | vb_ref | 사용하는 VertexBuffer 인덱스 |
| **0x08** | **bmap** | **사용하는 조인트 팔레트 인덱스** (0x10006의 몇 번째 팔레트) |
| 0x0C | matID(palID) | 머티리얼/palID |
| 0x14 | (mat 중복) | 머티리얼 중복 저장 |
| 0x18 | (mat 중복) | 머티리얼 중복 저장 |
| 0x1C | ib_ref | 사용하는 IndexBuffer 인덱스 |
| 0x28 | vb_start | VB 내 시작 정점 |
| 0x2C | numv | 정점 수 |
| 0x30 | ib_start | IB 내 시작 인덱스 |
| 0x34 | numi | 인덱스 수 |

- **머티리얼은 0x0C·0x14·0x18 세 곳에 중복 저장**된다. 렌더에 반영하려면 **셋 다** 바꿔야 한다(0x18만 바꾸면 무시됨). [실측]
- **서브메시를 렌더에서 끄려면 numi(0x34)=0**으로 만든다. [실측]

## 3. 조인트 팔레트 (0x10006) [실측]

**스키닝에 쓰이는 본 목록**을 팔레트 단위로 담는다.

### 3.1 구조
```
0x10006 inner:
  npal (u32)                     # 팔레트 개수
  반복 npal회:
    cnt (u32)                    # 이 팔레트의 엔트리 수
    반복 cnt회:
      (G1MMIndex u32, physicsIndex u32, jointIndex u32)   # 12바이트
```

### 3.2 엔트리 3필드의 의미 [실측]
- **G1MMIndex** — G1MM 역바인드행렬 색인. **실제 스키닝 행렬을 결정**한다.
- **physicsIndex** — **물리(클로스) 표식.** `≠0`이면 그 팔레트는 **클로스 물리 팔레트**로, 게임이 "이 팔레트를 쓰는 메시는 물리(클로스) 메시"라고 인식한다. `=0`이면 일반 스키닝 팔레트. (이것이 클로스 저작의 핵심 — [06]·[07].)
- **jointIndex** — 본 ID. **디스플레이 팔레트에서는 로컬 본**, **클로스 물리 팔레트에서는 글로벌 본**(= parentID)로 관측됨. [실측]

### 3.3 정점 → 팔레트 스키닝 [실측/유도]
- 정점의 블렌드 인덱스(레이아웃상 `i1` 필드)가 팔레트 슬롯을 가리킨다.
- **일반/리벳 정점**: `i1 = 3 × (팔레트 슬롯 인덱스)`로 인코딩된다(게임이 //3 해석). 슬롯 → G1MMIndex → 행렬. [실측]
  - 즉 리벳 정점의 `i1` 첫 바이트 = `3 × slot`.
- (**클로스 시뮬 정점**은 다르게 해석된다 — [06] §정점 시맨틱 참조.)

## 4. MeshGroups (0x10009) 와 ext_id [실측]

메시그룹은 서브메시를 그룹으로 묶고, **클로스/물리 연결(ext_id)**을 담는다.

### 4.1 구조 (관측)
- 헤더(블록 카운트들) 뒤에 엔트리들이 온다. 각 엔트리(가변 길이):
  - `@0x10` : 플래그(1=클로스 계열)
  - **`@0x14` : ext_id** (외부/물리 연결 ID)
  - `@0x18` : 뒤따르는 서브메시 인덱스 개수(idxcnt)
  - 그 뒤 `idxcnt`개의 서브메시 인덱스(u32)

### 4.2 ext_id 인코딩 [실측]
디스플레이 메시가 어느 물리(클로스) 엔트리에 연결되는지를 ext_id로 표현한다:

| ext_id 범위 | 연결 대상 | 산식 |
|---|---|---|
| 0 ~ 9999 | NUNO1 엔트리 | ext = 0·10000 + idx |
| 10000 ~ 19999 | NUNV 엔트리 | ext = 1·10000 + idx |
| **20000 ~ 29999** | **NUNO3 엔트리** | ext = 2·10000 + idx |
| 0xFFFFFFFF | (연결 없음, 일반 메시) | — |

- **클로스 디스플레이 메시는 NUNO3(ext=20000+idx)에 연결된다.** [실측] NUNO1에는 디스플레이 메시가 없다(물리 격자로만 존재 — [06]).

### 4.3 삽입 규칙 [실측]
- 새 클로스 디스플레이 엔트리는 **block1(sub0과 같은 티어)** 끝에 삽입해야 렌더된다. block2/다른 티어에 넣으면 렌더 컨텍스트가 달라 **안 보인다.** [실측]
- ext_id는 **20000 + NUNO3 인덱스**(시프트 없음).

## 5. VertexBuffer (0x10004) / Layout (0x10005) [실측]

- **0x10004**: `num_vb` 뒤에 각 VB가 `(0, stride, numv, 0)` 헤더 + raw 정점 데이터로 온다.
- **0x10005**: 각 VB에 대응하는 시맨틱 레이아웃(어떤 정점 속성이 어느 오프셋·데이터타입에 있는지). 정점버퍼를 해석하려면 레이아웃이 필요하다.
- 정점의 구체 시맨틱(위치/노멀/UV/블렌드/클로스 필드)은 레이아웃으로 정의되며, **클로스 정점의 특수 시맨틱**(cpW1/cpW2/i1 등)은 [06] §정점 시맨틱에서 다룬다.

## 6. 이 문서와 클로스의 접점

클로스 저작에서 반복 등장하는 개념을 미리 정리:

- **서브메시(0x10008)** : 클로스 디스플레이 메시. `bmap`(팔레트 인덱스)이 어느 팔레트를 쓰는지가 결정적.
- **조인트 팔레트(0x10006)** : `physicsIndex≠드라이버본`이어야 게임이 물리 메시로 인식. 아니면 스파이크.
- **메시그룹(0x10009)** : `ext_id=20000+NUNO3인덱스`로 디스플레이↔물리 연결.

이 세 가지가 어떻게 맞물려 클로스가 구동되는지는 다음 문서에서 전부 다룬다 → [06_g1m_cloth_system.md](06_g1m_cloth_system.md).
