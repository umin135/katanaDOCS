# 09. ktmod 기획안 (초기·불안정 — 참고용)

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].

> ## ⚠️ 이 문서 전체는 "초기 기획안"이며 불안정하다
> 아래 서술된 ktmod의 패키지 포맷·에디터·매니저·역할 분담은 **전부 초기 기획(design intent)에 불과하며, 확정 사양이 아니고 언제든 바뀔 수 있다.** 여기 적힌 대로 구현하면 안 된다. 특히:
> - **ktmod의 `Content/`(에셋 심볼릭 담기) 메커니즘은 무기한 연기(deferred indefinitely)로 결정되었다.** §3의 서술은 방향성 기록일 뿐, 추진 계획이 아니다.
> - 앞선 문서(01~08)의 **포맷 사실([실측])과 이 문서의 기획([가설])을 명확히 구분**해서 읽어라.

---

## 1. ktmod란 (의도) [가설]

- **ktmod** = 이 프로젝트가 구상한 커스텀 모드 패키지/저작 도구 체계.
- 목표: 모드 제작자가 **에셋(모델·텍스처·재질)과 간단한 매니페스트**만으로 모드를 만들 수 있게 한다. 엔진 내부 포맷([01]~[08])을 몰라도 되게.
- 대상: [01]의 RDB/RDX/FDATA·전략 B(리다이렉트) 위에서, 원본을 파괴하지 않고 에셋을 얹는다.

---

## 2. 패키지 포맷 (의도) [가설]

- **`.ktmod`** — ZIP 호환 컨테이너.
- 구성(초안):
  ```
  mod.json          # 매니페스트 (모드 이름/버전/대상 게임/에셋 목록)
  assets/           # 실제 에셋(또는 심볼릭 참조 — §3)
  ```
- 매니저가 `.ktmod`를 읽어 전략 B(RDB 엔트리 리다이렉트)로 게임에 적용.

---

## 3. Content/ 심볼릭 설계 (무기한 연기) [가설·연기됨]

> **이 절의 메커니즘은 무기한 연기되었다.** 기록으로만 남긴다.

- 아이디어: 해시로 참조되는 에셋(다른 에셋을 KTID/objID로 가리키는 것)은 패키지 안에 **항상 심볼릭 JSON(`@name.ext` 형식)**으로 담고, **매니저가 설치 시점에 objID/FileKtid로 해석**한다.
- `Content_Legacy` = 심볼릭 없이 raw 리다이렉트(단순 교체).
- 연기 이유(정황): objID→에셋 해석이 **게임 싱글톤 DB에 의존**([02] §5)해 오프라인 자동 해석이 어렵고, 네임해시([01] §5)도 미해결이라 심볼릭 해석 체계가 불안정.

---

## 4. 3계층 역할 분담 (의도) [가설]

저작을 세 층으로 나누려는 구상:

| 계층 | 역할(의도) |
|---|---|
| **애드온(3D 툴, 예 Blender)** | g1m(메시/스켈레톤/물리) + 지오 사이드카(grp) + 재질 슬롯(mtl) 저작. **Blender는 슬롯 갯수/인덱스만 다루고**, meshType→sid 미러. |
| **에디터(전용 GUI)** | 머티리얼 인스턴스(= 셰이더 타입 + 텍스처 배치) 편집. 프로젝트(폴더 기반, `project.ktproj`) 관리 + Build→`.ktmod`. |
| **매니저** | sid + kts/mpr/g1t 접착(gluing) + RDB 리다이렉트로 게임 적용. |

- 애드온은 "슬롯 갯수/인덱스"만, 실제 재질/텍스처 접착은 에디터·매니저가 담당한다는 분리. [가설]

---

## 5. 에디터/프로젝트/런처 (의도) [가설]

- **폴더 기반 프로젝트**(`project.ktproj`) + **Build → `.ktmod`**.
- **런처**: New / Open / Recent.
- UX 방향(초안): UE 스타일 도킹, 통합 Content 브라우저, 텍스처는 TGA/DDS.
- 저작 범위(초안): **DOA6LR만 완전 저작 지원**, 타 게임은 Legacy(단순 교체)만.

---

## 6. 구현 스택 (의도) [가설]

- **C# / .NET(LTS 8+) + Avalonia(11)**, 크로스플랫폼(Win/Linux), 단일 실행파일.
- 구조(초안): UI 무관 코어 + CLI + GUI + 테스트. GUI는 MVVM.
- 바이너리 처리: 리틀엔디안 span 기반 파싱, 대용량 .fdata는 메모리맵 고려.

---

## 7. 무엇이 확정이고 무엇이 아닌가

| 항목 | 상태 |
|---|---|
| RDB/RDX/FDATA 포맷·전략 B | **[실측] 확정** ([01]) |
| g1m 포맷·클로스 저작 | **[실측] 대부분 확정** ([03]~[07]) |
| **ktmod 패키지/에디터/역할분담** | **[가설] 초기 기획, 불안정** |
| **Content/ 심볼릭** | **무기한 연기** |
| 전체 자동 조립(sid+kts+g1t+rdb) | **미완** |

**정리:** 이 문서(09)의 내용은 "이런 방향으로 가려 했다"는 기록이다. 실제로 신뢰할 것은 [01]~[08]의 포맷 사실이며, 그 위에 어떤 도구를 얹을지는 열려 있다.
