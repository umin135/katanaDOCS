# 04. g1m 스켈레톤과 행렬 (로컬/글로벌 ID · G1MM · oid)

> 독립 문서. 외부 파일을 참조하지 않는다. 검증 태그: [실측]/[유도]/[가설]/[미해결].

이 문서는 g1m의 **본(bone) 좌표계**를 다룬다. 특히 **로컬 ID ↔ 글로벌 ID** 개념은 클로스 parentID를 이해하는 데 필수다. 클로스 저작이 자꾸 실패한 근본 원인 중 하나가 바로 이 ID 혼동이었다([06]·[07] 참조).

---

## 1. G1MS — 스켈레톤 청크 [실측]

G1MS는 본 계층(각 본의 부모, 로컬 변환)과 **jil(Joint Index List)**을 담는다.

### 1.1 본 데이터
- 각 본은 **부모 인덱스**와 **로컬 변환(Translation / RotationQuaternion / Scale)**을 가진다.
- 본의 **월드 변환**은 루트부터 부모를 따라 로컬 변환을 누적해 계산한다(`world[b] = world[parent] · local[b]`). [유도]

### 1.2 로컬 ID vs 글로벌 ID (★핵심)
g1m은 두 가지 본 번호 체계를 쓴다:

- **로컬 ID** — 이 g1m 내부 본 배열의 인덱스(0..N-1). "이 파일 안에서 몇 번째 본".
- **글로벌 ID** — 전역(병합/베이스) 스켈레톤에서의 본 ID. 여러 g1m(코스튬)이 공유하는 캐릭터 스켈레톤 기준 번호.

이 둘을 잇는 것이 **jil**이다.

---

## 2. jil — 로컬↔글로벌 매핑 [실측]

jil은 정수 배열이며, 다음 관계를 정의한다:

```
localIDToGlobalID[jil[i]] = i          # (표준 정의)
globalIDToLocalID[i]      = jil[i]
```

실무에서는 두 함수로 다룬다:

- **l2g(local) = local_to_gid(local)** : 로컬 본 → 글로벌 ID
- **g2l(global) = gid_to_local(global)** : 글로벌 ID → 로컬 본

### 2.1 jil 파싱 위치 [실측]
- jil의 개수(jic)는 G1MS inner의 **오프셋 0x0A(u16)**에 있다.
- jil 배열은 G1MS inner의 **오프셋 0x10**부터 `jic`개의 u16으로 온다.
  - (이 오프셋은 여러 번 잘못 잡았다가 실측으로 확정된 값이다. 원본 클로스의 parentID 해석이 일치하는지로 검증했다.)
- 무효 엔트리는 `0xFFFF`로 표시된다(해당 로컬 슬롯이 글로벌에 매핑 안 됨).

### 2.2 실측 예시 (한 코스튬 스켈레톤)
| 로컬 본 | 이름(oid) | l2g(글로벌) |
|---|---|---|
| bone10 | SK_Spine02 | 10 (**항등**) |
| bone68 | RF_L_ForeArm_01 | **75** |
| bone72 | RF_L_UpLeg_01 | **79** |

- **낮은 SK(주요 스켈레톤) 본은 로컬=글로벌(항등)인 경우가 많다.** (예: bone10→10)
- **RF(트위스트/리그) 본 등은 로컬≠글로벌**이다. (예: bone68 로컬 → 글로벌 75)
- 그래서 **"글로벌 75"를 로컬 번호로 착각하면 엉뚱한 본**(허벅지 등)을 가리키게 된다. parentID는 글로벌이므로 항상 **g2l을 적용**해 로컬 드라이버 본을 얻어야 한다. [실측] (이 혼동이 클로스 디버깅을 오래 헤매게 한 원인 중 하나.)

## 3. globalToFinal — 런타임 조인트 해석 [유도/가설]

게임 런타임은 글로벌 ID를 **최종 조인트 배열의 위치(final joint index)**로 매핑하는 **globalToFinal** 표를 만든다. 클로스 parentID(글로벌)는 이 표를 거쳐 실제 구동 조인트가 된다.

### 3.1 산출 방식 (관측/추론) [가설]
- 게임(및 참조 뷰어)은 스켈레톤 조인트를 순회하며 `globalToFinal[localIDToGlobalID[idx]] = jointIndex`로 채운다. jointIndex는 조인트가 추가되는 순서(0,1,2,…)다.
- 즉 **globalToFinal[글로벌] = 그 본이 최종 조인트 배열에서 갖는 위치**.

### 3.2 자기완결 g1m에서의 실측 관계 [실측]
- 커스텀 저작 대상이 되는 (거의) 자기완결(self-contained) g1m에서는, **`globalToFinal[l2g[X]] = X`(로컬 본 X로 되돌아옴)** 관계가 성립함이 실측으로 확인됐다.
  - 검증: 한 작동 클로스에서 `parentID=75=l2g[68]` → 게임 구동 본 = 로컬 bone68. `world[bone68]·CP` 위치가 그 클로스의 디스플레이 메시 위치와 **오차 0.6단위**로 일치. (상세 [06] forward model.)
- 이 관계가 **클로스 parentID는 반드시 글로벌(l2g[로컬])이어야 함**의 근거다. (자세한 유도는 [06]·[07].)

> **[미해결/주의]:** 외부 베이스 스켈레톤을 참조하는 "실제 게임 코스튬"에서 globalToFinal이 정확히 어떻게 병합되는지는 완전 검증하지 못했다. 우리 저작 대상 g1m에서는 위 관계가 성립했다.

## 4. G1MM — 역바인드행렬 [실측]

- G1MM은 **역바인드행렬(inverse bind matrix)** 4x4의 배열이다.
- **G1MMIndex**로 색인한다. G1MMIndex는 **로컬 본 순서의 순열(permutation)**이다(로컬 인덱스와 1:1 대응하되 순서가 다를 수 있음).
- 관계: **`inv(G1MM[G1MMIndex]) = world[bone]`** (역바인드의 역 = 그 본의 바인드 포즈 월드행렬). [실측]
  - 이 관계로, G1MM에서 본의 월드 바인드 행렬을 복원할 수 있다.
- 스키닝(정점 변형)은 이 G1MMIndex를 통해 본 행렬을 얻는다. 즉 **조인트 팔레트의 G1MMIndex 필드가 실제 스키닝 행렬을 결정**한다(→ [05]).

### 4.1 본의 월드 행렬 얻는 두 경로 [유도]
1. G1MS의 본 계층에서 부모 누적으로 직접 계산(`world[b]`).
2. G1MM에서 `inv(G1MM[G1MMIndex(b)])`.
   두 경로는 (바인드 포즈에서) 일치해야 한다. 검증에 유용.

## 5. oid — 본 이름 테이블 [실측]

`oid` 파일/데이터는 **글로벌 본 ID → 이름**을 준다(디버깅·해석에 필수).

### 5.1 구조 (type2) [실측]
```
oid:
  maxJointIndex (u32)                       # 최대 글로벌 인덱스
  반복: (global_idx u32, name_hash u32, test u32)   # 12바이트 삼중쌍
```
- 각 삼중쌍은 글로벌 본 인덱스와 **이름 해시**를 준다. 별도의 해시→문자열 테이블로 실제 이름을 얻는다.
- 예시 이름: `SK_Spine02`(척추), `RF_L_ForeArm_01`(왼 팔뚝), `RF_L_UpLeg_01`(왼 허벅지), `SK_L_Shoulder`, `SK_Hips`, `SK_L_Leg` 등.
  - **`SK_` 접두 = 주요 스켈레톤 본**, **`RF_` 접두 = 리그/트위스트(팔로우) 본**. [유도]

### 5.2 oid ↔ jil 정합 [실측]
- **oid의 유효 글로벌 집합 == jil의 유효 글로벌 집합**임이 확인됐다(같은 스켈레톤). [실측]
- 그래서 oid로 "글로벌 79 = RF_L_UpLeg_01" 같은 확인이 가능하고, 이는 jil의 l2g/g2l과 일치한다.

---

## 6. 실전 요약 (클로스 저작 관점)

- 클로스 **parentID는 글로벌 ID**다. **로컬 드라이버 본 = g2l(parentID)**.
- 새 클로스를 만들 때, 사용자가 고른 **로컬 드라이버 본 X**에 대해 **parentID = l2g(X)**로 써야 한다.
- **CP(제어점)는 `world[로컬 본 X]` 프레임에 인코딩**되고, 게임은 `world[globalToFinal[parentID]]=world[X]`로 배치 → **프레임 일치**. (→ [06] forward model이 이 전체를 검증한다.)
